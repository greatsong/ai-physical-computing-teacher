## 준비물 체크리스트

<div class="prep-checklist">
  <h5>하드웨어 (팀별)</h5>
  <ul>
    <li>프로젝트 시연용 컴퓨터 (대시보드·ML 모델 실행 가능 상태)</li>
    <li>Raspberry Pi Pico 2 WH + 팀별 센서 세트 (라이브 데모용)</li>
    <li>USB 데이터 케이블</li>
    <li>발표용 대형 모니터 또는 프로젝터 (교실 공용 1대)</li>
    <li>문서 카메라 (OLED 화면 확대 시연용, 있으면 활용)</li>
  </ul>
  <h5>소프트웨어 및 자료</h5>
  <ul>
    <li>팀별 대시보드 URL 또는 로컬 실행 파일</li>
    <li>백업용 스크린샷 또는 화면 녹화 영상 (라이브 데모 실패 대비 — 필수)</li>
    <li>피드백 양식 인쇄물 또는 디지털 양식 (3단계: 좋았던 점·궁금한 점·제안)</li>
    <li>타이머 앱 또는 스톱워치 (발표 시간 관리용)</li>
  </ul>
  <h5>수업 전 확인</h5>
  <ul>
    <li>프로젝터/모니터 HDMI 또는 USB-C 연결 테스트 완료</li>
    <li>각 팀 대시보드가 정상 작동하는지 사전 확인 — 안 되는 팀에 백업 방법 안내</li>
    <li>교실 와이파이 상태 확인 (웹 기반 대시보드·Teachable Machine 사용 팀)</li>
    <li>피드백 양식을 팀 수 × 학생 수만큼 인쇄하거나 디지털 링크 공유 완료</li>
    <li>발표 순서 결정 방법 준비 (제비뽑기 또는 자원 순서)</li>
    <li>각 팀 발표 사이 30초&#126;1분 전환 시간 확보 계획</li>
  </ul>
</div>

## 도입 (5분)

<div class="teacher-script">
  <p>"여러분, 지난 네 차시 동안 정말 대단한 일을 해왔어요. 문제를 직접 정의하고, 센서로 데이터를 수집하고, 대시보드로 시각화하고, ML 모델까지 학습시켰죠."</p>
  <p>"오늘은 그 <strong>모든 과정을 공유하는 날</strong>입니다. 발표는 단순히 결과물을 보여주는 게 아니에요. 여러분이 겪은 시행착오, 문제 해결 과정, 그리고 배운 점을 나누는 거예요."</p>
  <p>"엔지니어와 데이터 과학자에게 가장 중요한 능력 중 하나가 뭔지 아세요? <strong>자기가 만든 것을 설명하는 능력</strong>이에요. 아무리 좋은 시스템을 만들어도, 왜 이게 필요하고 어떻게 작동하는지 설명 못 하면 아무도 쓰지 않거든요."</p>
  <p>"그리고 하나 더 — 오늘 여러분은 <strong>발표자</strong>이면서 동시에 <strong>피드백 제공자</strong>입니다. 다른 팀의 발표를 듣고 구조화된 피드백을 작성할 거예요. 피드백은 세 가지입니다: <strong>좋았던 점</strong>, <strong>궁금한 점</strong>, <strong>제안</strong>. 이 세 가지를 기억하세요."</p>
</div>

<div class="student-voice">
  <span class="bubble">"발표 긴장돼요&#126;"</span>
  <span class="bubble">"라이브 데모가 잘 될까&#126;"</span>
  <span class="bubble">"다른 팀은 뭘 만들었을까?"</span>
  <span class="bubble">"피드백은 어떻게 쓰는 거예요?"</span>
</div>

<div class="teaching-tip">
  <p>발표에 대한 긴장감을 줄여주는 것이 도입의 핵심입니다. "틀리거나 실패한 이야기가 오히려 더 값진 발표"라는 메시지를 전달하세요. "라이브 데모가 안 되면 스크린샷을 보여줘도 괜찮다"고 미리 안심시키면 학생들의 부담이 크게 줄어듭니다. 또한 피드백의 3단계 구조를 칠판에 적어두면 학생들이 발표를 들으며 바로 메모할 수 있습니다.</p>
</div>

## 전개 (35분)

<div class="step-header">
  <span class="step-num">1</span>
  <span class="step-title">발표 준비 및 최종 점검</span>
  <span class="step-time">5분</span>
</div>

<div class="teacher-script">
  <p>"지금부터 5분간 팀별로 발표를 최종 점검하세요. 확인할 것 세 가지입니다."</p>
  <p>"첫째, <strong>역할 분담</strong>. 누가 어떤 부분을 말할지 정했나요? 한 사람만 말하면 안 됩니다. 팀원 전원이 최소 한 파트씩 담당하세요. 예를 들어 '문제 정의는 A, 센서와 데이터 수집은 B, 대시보드 시연은 C, ML 결과는 D, 느낀 점은 A'처럼요."</p>
  <p>"둘째, <strong>데모 테스트</strong>. 대시보드와 ML 모델이 지금 이 순간 잘 작동하는지 확인하세요. 안 되면 백업 스크린샷이나 영상을 바로 꺼낼 수 있도록 준비해 두세요."</p>
  <p>"셋째, <strong>시간 체크</strong>. 발표는 팀당 5&#126;7분, 질의응답 2&#126;3분입니다. 시간이 넘어가면 제가 안내판을 보여줄 거예요."</p>
</div>

<div class="student-voice">
  <span class="bubble">"나는 문제 정의 부분 말할게"</span>
  <span class="bubble">"대시보드 켜져 있어?"</span>
  <span class="bubble">"백업 스크린샷 어디 저장했더라?"</span>
  <span class="bubble">"한번 연습해보자, 시간 재볼게"</span>
</div>

<div class="trouble-cards">
  <div class="trouble-card">
    <div class="symptom">"대시보드가 갑자기 안 열려요!"</div>
    <span class="label">원인</span>
    <span class="value">서버가 꺼져 있거나 USB 연결 불량, 코드 미실행</span>
    <span class="label">대응</span>
    <span class="value">USB를 다시 꽂고 Thonny에서 코드를 재실행하세요. 그래도 안 되면 백업 스크린샷이나 화면 녹화 영상을 사용합니다. 발표 시 "원래는 이렇게 동작합니다"라고 설명하면 충분합니다</span>
  </div>
  <div class="trouble-card">
    <div class="symptom">"역할 분담이 안 돼요 — 한 명이 다 하겠대요"</div>
    <span class="label">원인</span>
    <span class="value">팀원 간 의사소통 부족이나 자신감 차이</span>
    <span class="label">대응</span>
    <span class="value">"문제 정의 → 데이터 수집 → 대시보드 시연 → ML 결과 → 느낀 점" 5파트로 명확히 나누어 각자 한 파트씩 담당하도록 안내하세요. 자신감이 부족한 학생에게는 "대시보드 시연"처럼 말보다 보여주기 중심인 파트를 맡기세요</span>
  </div>
</div>

<div class="patrol-point">
  <p><strong>이 시점에 순회하며 확인:</strong></p>
  <ol>
    <li>팀원 전원이 발표에서 맡은 역할이 명확한가</li>
    <li>라이브 데모가 작동하는가 — 안 되는 팀에 백업 방법을 안내했는가</li>
    <li>피드백 양식을 모든 학생이 받았는가</li>
    <li>발표 순서가 결정되었는가 (제비뽑기 또는 자원)</li>
    <li>긴장하는 학생에게 격려의 한마디를 건넸는가</li>
  </ol>
</div>

<div class="teaching-tip">
  <p>발표 순서는 자원하는 팀부터 시작하되, 자원 팀이 없으면 제비뽑기로 정하세요. 첫 번째 팀의 발표가 분위기를 결정하므로, 준비가 잘 된 팀이 먼저 하면 전체 수업의 질이 올라갑니다. 준비 시간이 5분이면 짧게 느껴질 수 있지만, 11&#126;13차시에서 이미 프로젝트를 완성했으므로 최종 점검 수준이면 충분합니다. 이 시간에 칠판에 "피드백 3단계: 좋았던 점 / 궁금한 점 / 제안"과 예시를 적어두세요.</p>
</div>

<div class="step-header">
  <span class="step-num">2</span>
  <span class="step-title">팀 발표 및 질의응답</span>
  <span class="step-time">25분</span>
</div>

<div class="teacher-script">
  <p>"자, 첫 번째 팀 나와주세요! 발표 구성을 다시 한번 확인합니다."</p>
  <p>"<strong>1단계</strong>: 문제 정의 — 어떤 문제를 왜 해결하려 했는지. <strong>2단계</strong>: 센서와 데이터 — 어떤 센서로 어떤 데이터를 모았는지. <strong>3단계</strong>: 대시보드 시연 — 실제로 작동하는 모습을 보여주세요. <strong>4단계</strong>: ML 결과 — 머신러닝 모델의 성능과 한계. <strong>5단계</strong>: 배운 점과 어려웠던 점."</p>
  <p>"청중 여러분, 발표를 들으면서 피드백 양식에 메모하세요. <strong>좋았던 점</strong>, <strong>궁금한 점</strong>, <strong>제안</strong> — 이 세 가지를 구체적으로 적어주세요."</p>
  <p>"발표 5&#126;7분, 질의응답 2&#126;3분입니다. 타이머 시작! 첫 번째 팀, 시작하세요!"</p>
</div>

<div class="student-voice">
  <span class="bubble">"저희 팀은 교실 환경 모니터링 프로젝트입니다"</span>
  <span class="bubble">"온도·습도·CO2 센서를 사용했고요&#126;"</span>
  <span class="bubble">"지금 대시보드를 보여드리겠습니다"</span>
  <span class="bubble">"ML로 환기 필요 여부를 자동 판단했어요"</span>
  <span class="bubble">"처음에 센서 연결이 안 돼서 한 시간이나 걸렸는데&#126;"</span>
  <span class="bubble">"정확도가 처음엔 62%였는데 데이터를 더 모으니까 85%로 올라갔어요"</span>
</div>

<div class="teacher-script">
  <p><em>(각 팀 발표 후)</em></p>
  <p>"감사합니다! 질문 있는 사람?"</p>
  <p><em>(질문이 없을 때)</em> "제가 하나 물어볼게요. 이 프로젝트에서 <strong>가장 어려웠던 기술적 문제</strong>는 뭐였어요? 그걸 어떻게 해결했는지 들려주세요."</p>
  <p><em>(질문이 모호할 때)</em> "좋은 질문이에요. 조금 더 구체적으로 물어볼까요? 예를 들어 '정확도가 낮았을 때 어떤 방법을 시도했는지', '데이터를 얼마나 모았는지' 이런 식으로요."</p>
  <p><em>(팀 전환 시)</em> "수고했어요! 30초 드릴 테니 다음 팀 셋업 준비하세요. 청중은 방금 팀 피드백을 마무리해 주세요."</p>
</div>

<div class="student-voice">
  <span class="bubble">"그 센서 데이터 얼마나 모았어요?"</span>
  <span class="bubble">"ML 정확도가 몇 퍼센트였어요?"</span>
  <span class="bubble">"대시보드 색상은 어떻게 정한 거예요?"</span>
  <span class="bubble">"실시간으로 데이터가 바뀌는 건가요?"</span>
  <span class="bubble">"센서가 고장 나면 어떻게 해요?"</span>
</div>

<div class="trouble-cards">
  <div class="trouble-card">
    <div class="symptom">라이브 데모가 갑자기 안 됨</div>
    <span class="label">원인</span>
    <span class="value">센서 연결 끊김, 네트워크 불안정, 서버 오류 등</span>
    <span class="label">대응</span>
    <span class="value">당황하지 않도록 "기술 데모에서 이런 일은 흔합니다"라고 말해주세요. 백업 스크린샷으로 전환하고, "원래는 이렇게 동작합니다"라고 설명하면 됩니다. 오히려 "실제 개발에서도 이런 상황이 자주 발생하죠?"라고 학습 기회로 활용하세요</span>
  </div>
  <div class="trouble-card">
    <div class="symptom">발표 시간이 너무 짧음 (2분 만에 끝남)</div>
    <span class="label">원인</span>
    <span class="value">발표 내용이 부족하거나 긴장으로 빨리 끝냄</span>
    <span class="label">대응</span>
    <span class="value">"혹시 데이터를 수집할 때 어려웠던 점이 있었나요?", "대시보드에서 가장 신경 쓴 부분이 뭐예요?", "다시 한다면 바꾸고 싶은 점이 있나요?" 같은 후속 질문으로 보충하세요</span>
  </div>
  <div class="trouble-card">
    <div class="symptom">발표 시간이 너무 김 (7분 초과)</div>
    <span class="label">원인</span>
    <span class="value">시간 관리 부족, 내용이 많아 줄이지 못함</span>
    <span class="label">대응</span>
    <span class="value">5분 경과 시 "1분 남았습니다" 안내판을 보여주세요. 6분에 "마무리해 주세요" 신호를 보내고, 7분이 넘어가면 부드럽게 "핵심만 한 문장으로 마무리해 볼까요?"라고 안내합니다</span>
  </div>
  <div class="trouble-card">
    <div class="symptom">질의응답 시간에 아무도 질문을 안 함</div>
    <span class="label">원인</span>
    <span class="value">질문에 대한 부담감, 무엇을 물어야 할지 모름</span>
    <span class="label">대응</span>
    <span class="value">교사가 먼저 질문을 하나 던져 분위기를 만드세요. "가장 어려웠던 기술적 문제가 뭐였어요?", "ML 정확도를 더 높이려면 어떻게 해야 할까요?", "이 프로젝트를 실제로 학교에 설치한다면?" 같은 질문이 효과적입니다</span>
  </div>
  <div class="trouble-card">
    <div class="symptom">피드백에 "좋았어요"만 적음</div>
    <span class="label">원인</span>
    <span class="value">구체적 피드백 작성 경험 부족</span>
    <span class="label">대응</span>
    <span class="value">칠판에 피드백 예시를 적어주세요. 좋았던 점: "대시보드에서 색상으로 위험 수준을 보여준 점이 직관적이었다" / 궁금한 점: "센서 데이터를 몇 초 간격으로 수집했는지" / 제안: "알림 기능을 추가하면 더 실용적일 것 같다"</span>
  </div>
</div>

<div class="patrol-point">
  <p><strong>발표 진행 중 확인:</strong></p>
  <ol>
    <li>발표 팀의 팀원 전원이 발언 기회를 가지고 있는가</li>
    <li>청중이 피드백 양식에 실제로 기록하고 있는가 (빈 양식 없는지 확인)</li>
    <li>시간 관리가 되고 있는가 — 팀당 7&#126;10분(발표 + Q&A) 이내</li>
    <li>질의응답이 건설적으로 진행되고 있는가</li>
    <li>남은 팀 수와 남은 시간을 계산하여 속도 조절 — 4팀 기준 25분, 5팀이면 팀당 5분으로 압축</li>
  </ol>
</div>

<div class="teaching-tip">
  <p>팀 수에 따라 시간을 유연하게 조절하세요. 4팀이면 팀당 약 6분(발표 4분 + Q&A 2분), 5팀이면 팀당 약 5분(발표 3분 + Q&A 2분)으로 배분합니다. 시간이 부족할 경우, 질의응답을 1분으로 줄이고 나머지는 서면 피드백으로 대체하세요. 각 팀 발표 사이에 30초 전환 시간을 두면 다음 팀이 USB를 연결하고 코드를 실행할 여유가 생깁니다. 발표 중 학생이 "어려웠던 점"을 말할 때 교사가 "그걸 어떻게 해결했어요?"라고 한 마디만 추가하면 발표의 깊이가 확 달라집니다. 발표 내용을 국어 시간의 논증 구조(주장-근거-반박)와 연결하면 교과 통합 효과가 있습니다.</p>
</div>

<div class="step-header">
  <span class="step-num">3</span>
  <span class="step-title">피드백 정리 및 교환</span>
  <span class="step-time">5분</span>
</div>

<div class="teacher-script">
  <p>"모든 팀의 발표가 끝났습니다. 정말 수고했어요!"</p>
  <p>"이제 피드백 양식을 마무리할 시간이에요. 아직 다 못 적은 사람은 지금 완성하세요."</p>
  <p>"다시 한번 확인합니다. 피드백 세 가지: <strong>좋았던 점</strong>은 구체적으로 무엇이 왜 좋았는지, <strong>궁금한 점</strong>은 기술적 내용 위주로, <strong>제안</strong>은 실현 가능한 개선 아이디어로 적어주세요."</p>
  <p>"'좋았어요'는 피드백이 아닙니다. '대시보드에서 실시간 그래프가 자동으로 갱신되는 점이 인상적이었다'처럼 <strong>무엇이 왜 좋았는지</strong>를 적어야 해요."</p>
  <p>"작성이 끝나면 해당 팀에게 전달하세요. 그리고 받은 피드백을 팀별로 1분간 읽어보는 시간을 가질게요."</p>
</div>

<div class="student-voice">
  <span class="bubble">"구체적으로 쓰려니까 생각보다 어렵네&#126;"</span>
  <span class="bubble">"궁금한 점이 진짜 많았어요"</span>
  <span class="bubble">"저 팀 대시보드 디자인이 진짜 좋았는데"</span>
  <span class="bubble">"제안은 뭘 쓰지&#126; 알림 기능?"</span>
  <span class="bubble">"우리 팀 피드백 읽어보자!"</span>
</div>

<div class="patrol-point">
  <p><strong>순회 시 확인:</strong></p>
  <ol>
    <li>피드백이 세 가지 항목(좋았던 점·궁금한 점·제안)을 모두 포함하는가</li>
    <li>"좋았어요", "잘했어요" 같은 막연한 피드백 대신 구체적 내용이 있는가</li>
    <li>자기 팀을 제외한 모든 팀에 대해 피드백을 작성했는가</li>
    <li>피드백이 해당 팀에게 전달되었는가</li>
    <li>받은 피드백을 팀원들이 함께 읽고 있는가</li>
  </ol>
</div>

<div class="teaching-tip">
  <p>피드백 작성이 어려운 학생에게는 문장 시작 틀을 제공하세요. "&#126;&#126;이/가 인상적이었다, 왜냐하면 &#126;&#126;", "&#126;&#126;에 대해 더 알고 싶다", "&#126;&#126;을/를 추가하면 &#126;&#126;할 것 같다" 같은 틀을 칠판에 적어두면 학생들이 훨씬 쉽게 작성합니다. 피드백 교환 후 각 팀이 받은 피드백을 읽어보는 시간은 짧더라도 반드시 가지세요. "내 발표가 다른 사람에게 어떻게 보였는지"를 바로 확인하는 경험이 의사소통 역량 발달에 매우 효과적입니다.</p>
</div>

## 정리 (10분)

<div class="teacher-script">
  <p>"오늘 발표를 들으면서 어떤 생각이 들었어요?"</p>
  <p>"같은 센서를 써도 문제 정의에 따라 프로젝트가 완전히 달라졌죠? 어떤 팀은 교실 환경을 모니터링했고, 어떤 팀은 식물 관리에 활용했어요. <strong>기술은 같지만, 문제를 어떻게 정의하느냐에 따라 결과가 달라진다</strong> — 이것이 오늘의 가장 큰 교훈입니다."</p>
  <p>"발표 과정에서 중요한 것 하나 더 느꼈을 거예요. '데모가 안 됐는데 스크린샷으로 보여줬다', '처음엔 정확도가 낮았는데 데이터를 더 모았다'고 이야기한 팀이 있었잖아요. <strong>시행착오를 솔직하게 공유한 것</strong>이 가장 값진 발표였어요."</p>
  <p>"받은 피드백 중에서 가장 인상적인 것을 팀별로 하나씩 공유해볼까요? '이 피드백이 특히 도움이 됐다'고 생각하는 걸 말해보세요."</p>
</div>

<div class="student-voice">
  <span class="bubble">"다른 팀 발표 보니까 우리도 저렇게 할 걸&#126;"</span>
  <span class="bubble">"피드백 받으니까 개선할 점이 보여요"</span>
  <span class="bubble">"발표하면서 오히려 정리가 된 것 같아요"</span>
  <span class="bubble">"ML 정확도를 더 올리고 싶어요"</span>
  <span class="bubble">"알림 기능 제안이 좋았어요"</span>
</div>

<div class="teacher-script">
  <p>"여러분은 액트3에서 정말 대단한 일을 해냈어요. <strong>문제를 정의하고, 센서로 데이터를 수집하고, 대시보드로 시각화하고, ML로 자동 판단까지</strong> — 그리고 오늘 그 과정을 다른 사람에게 설명하고 피드백을 주고받았어요. 이게 바로 <strong>AI 피지컬 컴퓨팅의 전체 사이클</strong>입니다."</p>
  <p>"스타트업이 새로운 서비스를 만들 때, 데이터 과학자가 분석 결과를 보고할 때 — 모두 '문제→프로토타입→데이터→발표→피드백'을 반복합니다. 여러분은 이 사이클을 네 차시에 걸쳐 직접 경험한 거예요."</p>
  <p>"그리고 피드백도 결국 <strong>데이터</strong>예요. 여러 사람의 의견을 모으고, 패턴을 찾고, 개선 방향을 정하는 것 — 센서 데이터를 분석하는 것과 같은 원리죠."</p>
  <p>"마지막으로, 이 프로젝트를 한 문장으로 요약해 봅시다. '우리 팀은 &#126;&#126; 문제를 &#126;&#126; 센서와 &#126;&#126; 기술로 해결했다.' 이 한 문장을 팀별로 만들어 보세요."</p>
  <p>"다음 시간은 코스 A의 <strong>마지막 차시</strong>입니다. 15주 동안의 여정을 되돌아보고, 얼마나 성장했는지 확인해볼 거예요. 수고하셨습니다!"</p>
</div>

<div class="teaching-tip">
  <p>정리 시간이 10분으로 넉넉하므로, 각 팀이 받은 피드백 중 가장 인상적인 것을 하나씩 공유하는 시간을 반드시 가지세요. "어떤 피드백이 가장 도움이 됐어요?"라고 물으면, 피드백을 주는 쪽과 받는 쪽 모두 "좋은 피드백이란 무엇인가"를 성찰할 수 있습니다. 또한 모든 팀의 노력을 인정해 주세요 — "이 팀은 대시보드 디자인이 뛰어나고, 저 팀은 데이터 분석이 깊었고, 이 팀은 ML 활용이 창의적이었다" 처럼 각 팀만의 강점을 한 가지씩 꼭 언급하세요. 완성도가 낮은 팀에게는 "도전한 것 자체가 대단한 것"이라고 격려하고, 구체적으로 잘한 부분을 찾아 이야기해 주세요.</p>
</div>

## 차별화 전략

<div class="differentiation">
  <div class="fast-learner">
    <p><strong>빠른 학생에게:</strong></p>
    <ul>
      <li>발표에서 문제 정의부터 해결까지의 논리적 흐름(문제→가설→데이터→분석→결론)을 구성하고, 기술적 도전과 해결 과정을 깊이 있게 설명하도록 안내하세요</li>
      <li>다른 팀 발표에 대해 "왜 그 센서를 선택했는지", "ML 모델의 한계를 어떻게 극복할 수 있는지", "실제 배포 시 고려할 점" 등 심화 질문을 하도록 유도하세요</li>
      <li>피드백에서 단순한 칭찬이 아닌, 기술적 개선점이나 확장 아이디어를 제안하도록 격려하세요</li>
      <li>전체 토론을 이끄는 역할(사회자, 타임키퍼)을 맡기면 리더십 역량도 기를 수 있습니다</li>
      <li>라이브 시연 중 예상치 못한 데이터 변동을 즉석에서 해석하는 연습을 해보도록 하세요</li>
    </ul>
  </div>
  <div class="slow-learner">
    <p><strong>느린 학생에게:</strong></p>
    <ul>
      <li>발표에서 자신이 담당한 한 가지 역할(예: 센서 연결, 데이터 수집)만 설명해도 충분합니다</li>
      <li>발표 시 대본을 보고 읽어도 괜찮다고 안심시키세요</li>
      <li>피드백 양식에 문장 시작 틀("&#126;&#126;이 좋았다, 왜냐하면 &#126;&#126;")을 미리 적어주세요</li>
      <li>피드백 작성이 어려우면 "가장 기억에 남는 장면 한 가지"만 적어도 됩니다</li>
      <li>말하기가 어려운 학생에게는 대시보드 시연(버튼 누르기, 화면 전환)처럼 보여주기 중심 역할을 맡기세요</li>
    </ul>
  </div>
</div>
