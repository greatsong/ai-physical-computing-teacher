## 1. 코스 B 개요

코스 B는 코스 A(15차시)를 통해 기초를 다진 학생들이 **자기 주도적으로 프로젝트를 수행**하는 심화 과정입니다. 교사가 시범을 보이고 학생이 따라하는 코스 A와 달리, 코스 B에서는 학생이 직접 센서를 선택하고, 시스템을 설계하고, 문제를 해결합니다.

### 구조

| 레벨 | 이름 | 차시 | 시간 | 핵심 키워드 |
|------|------|------|------|-------------|
| Lv.1 | 탐험가 | 3차시 | 약 150분 | 센서 1개 + OLED + CSV + 조건 판단 |
| Lv.2 | 발명가 | 3차시 | 약 150분 | 복합 센서 + Google Sheets 클라우드 |
| Lv.3 | 연구자 | 3차시 | 약 150분 | Flask 서버 + 실시간 대시보드 + AI 분석 |
| Lv.4 | 마스터 | 3차시 | 약 150분 | ML 분류 + Supabase + 종합 대시보드 |

### 운영 원칙

- **학생 주도**: 학생이 개별 또는 소그룹(2~3명)으로 프로젝트를 진행합니다.
- **자기 속도 학습**: 모든 학생이 같은 레벨을 동시에 할 필요가 없습니다. 빠른 학생은 Lv.4까지, 느린 학생은 Lv.2까지만 해도 괜찮습니다.
- **레벨 순차 진행**: 반드시 Lv.1부터 시작합니다. 각 레벨이 다음 레벨의 기반이 됩니다.
- **산출물 중심**: 매 레벨마다 동작하는 결과물이 나와야 합니다.

## 2. 교사 역할 변화

### 코스 A vs 코스 B 비교

| 구분 | 코스 A | 코스 B |
|------|--------|--------|
| 교사 역할 | 강사 (시범 → 따라하기 → 코칭) | 퍼실리테이터 (질문 → 관찰 → 코칭) |
| 수업 흐름 | 전체가 같은 활동 | 개별/소그룹이 다른 활동 |
| 진도 관리 | 교사가 통제 | 학생이 자기 속도로 진행 |
| 에러 대응 | 교사가 직접 해결 | 학생이 먼저 시도 → 교사가 힌트 |

### 퍼실리테이터로서의 행동 지침

**개입해야 할 때:**
- 하드웨어 연결 오류로 부품이 손상될 위험이 있을 때 (예: 전원 핀 역접)
- 학생이 15분 이상 같은 문제에 멈춰 있고, 스스로 해결 전략을 시도하지 않을 때
- 안전 문제가 발생했을 때 (합선, 과열 등)
- 학생 간 갈등이 학습을 방해할 때

**기다려야 할 때:**
- 학생이 에러 메시지를 읽고 원인을 추론하고 있을 때
- 코드가 동작하지 않지만 학생이 디버깅 전략을 시도하고 있을 때
- 학생이 동료에게 도움을 요청하고 있을 때
- 완벽하지 않지만 자기만의 방식으로 문제를 해결하고 있을 때

### 수준별 학생 지원 전략

| 학생 유형 | 지원 방법 |
|----------|----------|
| 빠른 학생 | 다음 레벨로 자연스럽게 진행 / 동료 멘토 역할 부여 |
| 보통 학생 | 체크리스트 제공 + 정기적 진행 확인 |
| 느린 학생 | AI 앱(Pico Makers) 적극 활용 유도 + 1:1 코칭 시간 확보 |
| 코스 A 미이수 학생 | Lv.1에서 코스 A 핵심 개념 복습 시간 추가 배정 |

## 3. 레벨별 상세 지도 가이드

### Lv.1 탐험가 (3차시, 약 150분)

**목표**: 센서 1개를 선택하여 OLED 대시보드를 만들고, CSV 파일로 데이터를 저장하며, 조건 판단 로직을 구현합니다.

**센서 선택지**: DHT20(온습도), SOUND(소리), LIGHT(빛) 중 택 1

**핵심 역량**: `if/else` 조건문 활용, OLED 대시보드 레이아웃 설계, 파일 입출력

#### 차시별 운영 제안

| 차시 | 시간 | 활동 |
|------|------|------|
| 1차시 | 50분 | 센서 선택 + 배선 + 기본 데이터 읽기 확인 |
| 2차시 | 50분 | OLED 대시보드 레이아웃 설계 + 구현 |
| 3차시 | 50분 | CSV 저장 + 조건 판단 로직 + 테스트 + 발표 |

#### 교사 지도 포인트

**센서 선택 안내:**
- 센서 선택을 학생에게 맡기되, 각 센서의 난이도 차이를 안내합니다.
- DHT20(온습도)이 가장 표준적이고 안정적입니다. 처음 프로젝트를 하는 학생에게 추천하세요.
- SOUND(소리)와 LIGHT(빛)는 ADC 읽기 방식으로, 값의 변동이 크므로 평균 필터링이 필요할 수 있습니다.

**OLED 레이아웃:**
- 자유도를 주되, 128x64 픽셀 좌표계를 미리 설명하세요.
- 좌표(0,0)이 좌상단이고, `text("문자열", x, y)`로 출력한다는 점을 알려주세요.
- 모눈종이에 레이아웃을 먼저 스케치하게 하면 효과적입니다.

**CSV 저장:**
- `os.stat()` 헤더 중복 방지 패턴을 안내하세요. 파일이 이미 존재하면 헤더를 건너뛰고, 없으면 헤더를 먼저 쓰는 방식입니다.
- 파일명에 날짜를 넣는 습관을 들이게 하면 데이터 관리가 쉬워집니다.

**I2C 설정 (중요):**
- `I2C(1, sda=Pin(6), scl=Pin(7))` — 반드시 **I2C1 버스**를 사용합니다.
- I2C0이 아닌 I2C1인 이유를 물어보는 학생에게는 보드의 핀 배치도를 보여주세요.

#### 예상 학생 반응과 대응

| 학생 반응 | 교사 대응 |
|----------|----------|
| "어떤 센서 고르지?" | 관심 분야 기반으로 선택 유도 — 체육 좋아하면 소리(박수 카운터), 과학 좋아하면 온습도(교실 환경 모니터링), 미술 좋아하면 빛(조명 센서) |
| "조건문 어떻게 쓰지?" | 코스 A 4차시 내용 복습 유도. "코스 A에서 LED 켜고 끄는 조건 어떻게 만들었는지 떠올려 봐" |
| "OLED에 한글이 안 나와요" | OLED 라이브러리는 기본적으로 영문만 지원. 영문 + 숫자 + 아이콘으로 디자인하도록 안내 |
| "CSV 파일이 안 보여요" | Thonny에서 Pico 파일 탐색기 확인. `open("data.csv", "a")` 에서 파일 경로 확인 |

#### 평가 기준

| 등급 | 기준 |
|------|------|
| **B** (기본) | 대시보드에 실시간 센서값 + 상태 표시 + CSV 저장이 동작함 |
| **A** (우수) | 스파크라인, 막대그래프 등 시각적 요소를 자발적으로 추가함 |
| **S** (탁월) | 디자인 원리를 설명할 수 있고, 다른 센서를 추가로 연동 시도함 |

### Lv.2 발명가 (3차시, 약 150분)

**목표**: 복합 센서(2~3개)를 조합하고, Google Sheets에 실시간 데이터를 클라우드 저장합니다.

**WiFi 설정**: Pico 2 WH 내장 WiFi, `network.WLAN(network.STA_IF)` 사용

**핵심 역량**: 네트워크 통신, HTTP POST, 데이터 클라우드 저장

#### 차시별 운영 제안

| 차시 | 시간 | 활동 |
|------|------|------|
| 1차시 | 50분 | WiFi 연결 + Google Sheets 설정 + Apps Script 배포 |
| 2차시 | 50분 | 복합 센서 배선 + 데이터 수집 코드 작성 |
| 3차시 | 50분 | HTTP POST 전송 + 테스트 + 데이터 분석 + 발표 |

#### 교사 지도 포인트

**WiFi 연결 (가장 큰 허들):**
- 학교 WiFi가 **2.4GHz**를 지원하는지 반드시 사전 확인하세요. Pico W는 5GHz를 지원하지 않습니다.
- 많은 학교 WiFi는 802.1X 인증(아이디/비밀번호)을 사용하는데, Pico W에서는 이 인증을 지원하지 않습니다.
- **핫스팟 백업 방법**: 학생 스마트폰 테더링을 대안으로 안내하세요. 핫스팟 이름과 비밀번호를 단순하게 설정하도록 합니다.
- WiFi 연결 코드에 재시도 로직(`while not wlan.isconnected()`)을 반드시 포함하세요.

**Google Apps Script 설정 (주의사항):**
- Apps Script는 반드시 **Sheets에서 Extensions > Apps Script**로 생성해야 합니다. 독립 실행형(script.google.com에서 직접 생성)은 `getActiveSpreadsheet()`가 실패합니다.
- 배포 시 **Deploy > New deployment > Web app > Anyone** 설정이 필요합니다.
- 배포 URL을 복사할 때, `/exec`으로 끝나는 전체 URL을 사용해야 합니다.

**urequests 리다이렉트 문제:**
- MicroPython의 `urequests`는 302 리다이렉트를 자동으로 따라가지 않습니다.
- 리다이렉트 처리 코드를 별도로 작성해야 합니다. 이 부분은 Pico Makers 앱이 자동으로 생성해 주므로, 앱 활용을 권장합니다.

**예상 소요 시간:**
- WiFi 세팅: 30분 (문제 발생 시 더 걸릴 수 있음)
- Apps Script 설정: 20분
- 코드 작성: 50분
- 테스트 및 디버깅: 50분

#### 예상 학생 반응과 대응

| 학생 반응 | 교사 대응 |
|----------|----------|
| "WiFi가 연결이 안 돼요" | 2.4GHz 확인 → 핫스팟 전환 → SSID/비밀번호 오타 확인 |
| "Apps Script 배포가 안 돼요" | Google 계정 로그인 상태 확인 → 독립 실행형이 아닌 Sheets 연결형인지 확인 |
| "데이터가 Sheets에 안 올라가요" | 시리얼 모니터에서 HTTP 응답 코드 확인 → 302면 리다이렉트 처리 코드 추가 |
| "센서 2개를 어떻게 동시에 읽어요?" | I2C 센서는 주소가 다르면 같은 버스에서 동시 사용 가능, ADC 센서는 핀만 다르면 됨 |

#### 평가 기준

| 등급 | 기준 |
|------|------|
| **B** (기본) | 2개 이상 센서 데이터가 Google Sheets에 실시간 기록됨 |
| **A** (우수) | 조건 판단 로직(AND/OR)이 정확하게 동작함 |
| **S** (탁월) | Sheets에서 분석용 수식이나 차트를 자발적으로 추가함 |

### Lv.3 연구자 (3차시, 약 150분)

**목표**: 장기 데이터를 수집하고, PC에서 Flask 서버를 실행하여 실시간 대시보드를 구성하며, AI 분석과 연계합니다.

**핵심 역량**: 클라이언트-서버 통신 이해, 실시간 데이터 스트리밍, 데이터 시각화

#### 차시별 운영 제안

| 차시 | 시간 | 활동 |
|------|------|------|
| 1차시 | 50분 | Flask 설치 + 서버 코드 작성 + 기본 라우트 확인 |
| 2차시 | 50분 | Pico → Flask JSON POST + 실시간 그래프 구현 |
| 3차시 | 50분 | 장기 데이터 수집 + AI 분석 연계 + 연구 보고서 작성 |

#### 교사 지도 포인트

**Flask 서버 환경 설정:**
- Flask 서버는 **같은 WiFi 네트워크** 안에서만 동작합니다. Pico와 PC가 같은 네트워크에 있어야 합니다.
- PC의 IP 주소 찾기: Windows는 `ipconfig`, Mac은 `시스템 설정 > Wi-Fi > 세부사항`에서 확인합니다.
- Python이 설치되어 있어야 합니다. `pip install flask` 로 Flask를 설치하세요.
- `app.run(host='0.0.0.0', port=5000)`으로 실행해야 외부에서 접근 가능합니다 (`localhost`만 쓰면 외부 접근 불가).

**Pico → Flask 통신:**
- Pico에서 `urequests.post()`로 JSON 데이터를 Flask 서버에 전송합니다.
- Flask 서버의 엔드포인트 URL에 PC의 실제 IP 주소를 사용해야 합니다 (`http://192.168.x.x:5000/data`).
- 데이터 전송 간격은 5~10초가 적당합니다. 너무 짧으면 네트워크 부하, 너무 길면 실시간성이 떨어집니다.

**I2C 설정:**
- `I2C(1, sda=Pin(6), scl=Pin(7))` — I2C1 버스를 사용합니다.

**실시간 그래프:**
- matplotlib의 `FuncAnimation`을 사용하면 실시간 업데이트 그래프를 만들 수 있습니다.
- 웹 기반 대시보드를 원하면 Flask + Chart.js 조합도 가능합니다.

#### 예상 학생 반응과 대응

| 학생 반응 | 교사 대응 |
|----------|----------|
| "Flask가 뭐예요?" | "PC에서 실행하는 작은 웹 서버야. Pico가 데이터를 보내면 PC가 받아서 그래프로 보여주는 거지." |
| "서버가 안 켜져요" | Python 경로 확인 → 포트 충돌 확인 → 방화벽 설정 확인 |
| "Pico에서 데이터를 보내는데 서버에서 안 받아요" | IP 주소 확인 → 같은 네트워크인지 확인 → Flask 콘솔 로그 확인 |
| "AI 분석은 어떻게 해요?" | 수집된 데이터를 ChatGPT나 Claude에 붙여넣기하여 패턴 분석 요청 |

#### 평가 기준

| 등급 | 기준 |
|------|------|
| **B** (기본) | Pico 데이터가 PC 대시보드에 실시간 표시됨 |
| **A** (우수) | 데이터 분석과 AI 인사이트를 결합하여 의미 있는 발견을 도출함 |
| **S** (탁월) | 연구 보고서 형태로 결과를 체계적으로 정리하고 발표함 |

### Lv.4 마스터 (3차시, 약 150분)

**목표**: 규칙 기반 머신러닝 분류를 구현하고, Supabase 클라우드 데이터베이스에 데이터를 저장하며, 종합 대시보드를 완성합니다.

**핵심 역량**: 데이터 라벨링, 규칙 기반 분류, 클라우드 데이터베이스 활용, 종합 시스템 설계

#### 차시별 운영 제안

| 차시 | 시간 | 활동 |
|------|------|------|
| 1차시 | 50분 | Supabase 가입 + 테이블 생성 + REST API 연동 |
| 2차시 | 50분 | 데이터 라벨링 + 규칙 기반 분류 로직 구현 |
| 3차시 | 50분 | 종합 대시보드 + 분류 정확도 분석 + 최종 발표 |

#### 교사 지도 포인트

**Supabase 설정:**
- Supabase 무료 플랜에 가입합니다 (이메일 필요 — 학교 계정 사용을 권장합니다).
- 프로젝트 생성 후, 테이블을 만듭니다. Column 이름과 타입을 지도하세요:
  - `id` (int8, Primary Key, auto-increment)
  - `timestamp` (timestamptz, default: `now()`)
  - `temperature` (float4)
  - `humidity` (float4)
  - `co2` (float4) — 사용하는 센서에 따라 변경
  - `label` (text) — 분류 결과 저장용
- REST API의 URL과 anon key는 **Settings > API**에서 확인합니다.

**REST API 활용:**
- Pico에서 `urequests.post()`로 Supabase REST API에 직접 데이터를 전송합니다.
- 헤더에 `apikey`와 `Authorization: Bearer {anon_key}`를 포함해야 합니다.
- anon key는 코드에 하드코딩해도 무방합니다 (RLS 정책으로 보안 관리).

**규칙 기반 ML:**
- 머신러닝의 본질은 "데이터에서 패턴을 찾아 분류하는 것"이라고 설명하세요.
- 처음에는 `if/else`로 직접 규칙을 만들어 분류합니다 (예: 온도 > 28 AND 습도 > 70 → "불쾌").
- 이것이 "규칙 기반 분류기"이며, 이를 자동화한 것이 scikit-learn 같은 ML 라이브러리라고 연결하세요.
- 분류 정확도를 계산하여 "내 규칙이 얼마나 잘 맞는지"를 확인하게 하면 ML 개념 이해에 큰 도움이 됩니다.

**ADC 핀 매핑 (아날로그 센서):**
- GP26 = ADC0 (빛 센서)
- GP27 = ADC1 (소리 센서)
- GP28 = ADC2 (먼지 센서)

| "머신러닝이 너무 어려워요" | "지금 하고 있는 if/else 조건문이 사실 가장 기본적인 머신러닝이야!" |

- **속도가 아닌 깊이를 평가하세요.** Lv.2까지만 도달했더라도 깊이 있게 수행한 학생이 Lv.4를 대충 넘긴 학생보다 높은 평가를 받을 수 있습니다.

- **동료 평가를 활용하세요.** 학생끼리 서로의 프로젝트를 평가하는 시간을 가지면, 다양한 접근 방식을 배울 수 있습니다.
- **AI 도구 사용을 감점하지 마세요.** AI가 생성한 코드를 이해하고 수정할 수 있는 능력이 중요합니다. 코드를 그대로 복붙했는지, 이해하고 수정했는지를 구분하여 평가하세요.

## 6. 자주 묻는 질문

### 코스 운영

**Q: 코스 A를 안 하고 코스 B부터 해도 되나요?**

가능합니다. 다만 Lv.1에서 기초(브레드보드 배선, MicroPython 문법, Thonny 사용법)를 충분히 다지도록 시간을 넉넉하게 배분해야 합니다. 코스 A의 1~5차시 핵심 내용을 30분 정도 압축 안내한 뒤 Lv.1을 시작하는 것을 권장합니다.

**Q: 학생이 Lv.4까지 다 못 끝내면?**

Lv.2까지만 해도 충분히 의미 있는 학습입니다. 센서 데이터를 읽고, 조건 판단을 하고, 클라우드에 저장하는 것만으로도 IoT의 핵심 개념을 경험한 것입니다. 속도를 강요하지 마세요. 학생에게 "어디까지 도달했는지"가 아니라 "무엇을 배웠는지"를 물어보세요.

**Q: 한 반에서 학생들이 서로 다른 레벨을 하고 있으면 관리가 어렵지 않나요?**

처음에는 어려울 수 있지만, 디지털 도구(Pico Makers, Pico Buddy, 웹 교재)를 활용하면 학생이 상당 부분 자기 주도로 진행할 수 있습니다. 교사는 전체를 순회하며 코칭하고, 빠른 학생에게 동료 멘토 역할을 맡기면 효과적입니다.

### 환경 문제

**Q: WiFi가 안 되는 학교는 어떻게 하나요?**

두 가지 방법이 있습니다:
1. Lv.1(로컬 전용)만 수행합니다. OLED + CSV만으로도 충분히 의미 있는 프로젝트입니다.
2. 학생 스마트폰의 핫스팟(테더링)을 사용합니다. 데이터 요금이 걱정된다면, WiFi 연결 테스트만 핫스팟으로 하고 나머지는 로컬로 진행하세요.

**Q: 학교 컴퓨터에 Python/Flask를 설치할 수 없으면?**

Lv.3에서 Flask 대신 Google Sheets(Lv.2 방식)를 확장하여 사용할 수 있습니다. 또는 교사 PC 한 대에 Flask 서버를 설치하고, 여러 학생의 Pico가 동일 서버로 데이터를 전송하는 방식도 가능합니다.

### AI 도구 관련

**Q: AI 앱(Pico Makers)을 쓰면 학생이 너무 의존하지 않나요?**

AI가 만든 코드를 반드시 **검토하고 수정**하게 하는 것이 핵심입니다. 이것이 "바이브 코딩" 원칙입니다. 구체적인 방법:
- AI 코드를 받은 후, "이 코드에서 센서를 읽는 줄은 몇 번째 줄이야?"라고 물어보세요.
- "임계값을 25에서 30으로 바꿔 봐"처럼 코드 수정을 요구하세요.
- "이 코드가 왜 이렇게 동작하는지 설명해 봐"라고 물어보세요.
- 코드를 이해하지 못하고 그대로 사용하는 것과, 이해하고 수정하는 것의 차이를 강조하세요.

**Q: AI가 잘못된 코드를 생성하면 어떡하나요?**

오히려 좋은 학습 기회입니다. "AI가 만든 코드가 에러가 나는데, 어디가 잘못인지 찾아볼까?"라고 유도하세요. AI의 실수를 발견하고 수정하는 과정에서 학생의 코드 이해력이 크게 향상됩니다.

### 평가 관련

**Q: 코드를 못 짜는 학생은 어떻게 평가하나요?**

AI 도구를 활용하여 코드를 생성하되, 그 코드를 이해하고 설명할 수 있는지를 평가하세요. "코드를 직접 짜는 능력"이 아니라 "코드를 읽고 수정하고 활용하는 능력"이 평가 대상입니다. 이것은 AI 시대에 더 중요한 역량입니다.

**Q: 그룹 프로젝트에서 무임승차하는 학생이 있으면?**

각 학생에게 역할을 명확히 분담하세요 (예: 배선 담당, 코드 담당, 데이터 분석 담당). 개별 기여도를 확인하기 위해 짧은 개별 면담(1~2분)을 진행하거나, 각자의 학습 일지를 작성하게 하세요.
